（4）为登录列表做相应操作

思路：

​ 1.为登录表单项设置请求地址、请求方式以及表单项name属性

​ 2.当用户点击登录按钮时，客户端验证用户是否填写了登录表单

​ 3.如果其中一项没有输入，阻止表单提交

​ 4.服务器端接收请求参数，验证用户是否填写了登录表单

​ 5.如果其中一项没有输入，为客户端做出相应，阻止程序向下执行

​ 6.根据邮箱地址查询用户信息

​ 7.如果用户不存在，为客户端做出响应，阻止程序向下执行

​ 8.如果用户存在，将用户名和密码进行比对

​ 9.比对成功，用户登录成功

​ 10.比对失败，用户登录失败

逻辑：

​ 首先，用户需要来到登录页面，在登录表单中输入用户名和密码，点击登录按钮进行登录操作。

​ 用户登录操作需要将表单提交到服务器端，所以我要做的是为表单提交到服务器的地址（也就是实现登录功能的路由地址），还要为表单指定提交的方式。

​ 登录表单的提交方式一般采用post方式，因为post方式将请求参数（也就是用户填写的表单内容）放在请求体中进行反应，相对get方式会比较隐蔽一些。

​ get方式将表单填写的内容放置在了地址栏中间，想象在密码栏输入密码，点击登录提交。虽然在页面当中，密码框里的密码不是以明文进行展示的，但是一旦点击了提交按钮，密码会被立刻放置在地址栏中，这样别人就很容易看到你输入的密码是什么，不太安全。

​ 当用户点击提交按钮后，并不是立即将表单提交到服务器端。而是要对用户输入的内容进行验证，在用户登录表单中验证用户是否输入了邮件地址和密码，只有用户输入了才能进行提交操作。否则，就阻止表单提交。

​ 在登录表单中不需要验证邮箱的地址和格式以及密码的长度，因为正常的操作流程一定是用户先进行注册，然后再进行登录。在用户注册时才会验证邮箱的地址和格式以及密码的复杂度。

​ 在服务器端，我需要增加一个实现登录功能的路由地址，并指定路由地址的请求方式为post。然后开始接收用户通过表单传递过来的请求参数，接收完成以后，还要再次对请求参数进行验证。

​ 服务器端的验证是必不可少的一部分。为什么呢？因为客户端的javascript验证并不是100%安全的，在浏览器软件的设置当中，是可以禁用javascript代码。一旦客户端禁用了javascript代码,代码就不能执行了，客户端的验证就失效了。

​ 如果客户提交的请求参数通过了验证，接下来就要用邮箱地址查询用户信息。如果没有找到用户，就说明用户填写的邮箱地址是错误的。所以阻止程序向下运行，直接为客户端做出响应，告诉客户端邮箱地址或者密码错误。

​ 给客户端反馈的信息“客户端邮箱地址或者密码错误”。是为什么不直接告诉密码错误呢？因为如果有人恶意去猜，就降低了猜的难度。

​ 如果根据邮箱地址查询到了用户信息，就需要将用户在表单填写的密码和数据库中的相应密码进行比对，如果密码比对成功，说明用户登录成功。如果密码比对失败，就说明用户登录失败。

​ 无论是登录成功还是失败都要对客户端做出响应，否则，用户提交了表单，页面进行了跳转，但是却没有收到客户端的响应，客户端就会显示无法显示该页面了。

    //接收请求参数

    //如果用户没有输入邮箱地址
    if (email.trim().length == 0 || password.trim().length == 0){

        //没有查询到用户

（7）密码加密处理

​ 第三方模块bcrypt,使用哈希加密，哈希加密是单程加密方式：1234=>abcd。单程加密即只能加密不能解密。

​ 即使黑客侵入密码数据库也不能对密码进行解密，但是黑客还是有几率知道原始密码的。如何知道呢？虽然密码不能解密，但是黑客可以将一些常用的密码经过加密以后存储到数据库当中。在存储加密密码的同时，也存储了对应的明文密码。当黑客拿到我们数据库的加密密码以后，会拿着这个加密密码和黑客数据库中的加密密码进行比对。一旦比对成功，黑客就知道加密密码的明文是什么了。这种破解密码的方式叫暴力破解。

​ 为了增加暴力破解难度，在加密的密码中加入随机字符串可以增加密码被破解的难度

示例，属于异步API

//导入bcrypt模块
const bcrypt = require('bcrypt');
//生成随机字符串gen=>generate生成salt盐
let salt = await bcrypt.genSalt(10);
//使用随机字符串对密码进行加密
let pass = await bcrypt.hash('明文密码',salt);
//密码比对
let isEqual = await bcrypt.compare('明文密码'，'加密密码')；

​ 在服务器端登录成功后下一次再登录服务器端不认得客户端的原因，这是因为网站应用是基于http协议的，是基于请求和响应模型，这种命令有一个特点：在完成服务器端和客户端一次请求的响应后，客户端和服务器端就断了。也就是说，对于服务器端，它不关心客户端是什么，它只关心请求。请求来了，服务器端就响应，响应结束后客户端和服务器端就没有联系了。这个特点被称作http协议的状态性。

​ 要真正登录，要建立服务端和客户端的关联关系，建立这种联系要借助其它的技术cookie与session。在浏览器窗口的审查，Application->Cookies或者在Network->www.baidu.com->Request Headers->Cookies观察。

​ cookie:在客户端，浏览器在电脑硬盘中开辟的一块空间，主要供服务器端存储数据。

  • cookie中的数据是以域名的形式进行区分的。
  • cookie中的数据是有过期时间的，超过时间数据会被浏览器自动删除。
  • cookie中的数据会随着请求被自动发送到服务器端。

session:在服务器端，实际上就是一个对象，存储在服务器端的内存中，在session对象中也可以存储多条数据，每一条数据都有一个sessionid做为唯一标识。

​ 怎么实现登录拦截？当客户端向服务器端发送请求的时候，我们要判断下这个请求是不是以/admin开头的。如果这个请求是以/admin开头的，我们就判断一下用户的登录状态，如果用户是登录的，那么我让这个请求继续往下走。如果当前用户是没有登录的状态，我们就让这个页面重定向到登录页面，这样就可以了。因为登录页面也是/admin开头的，要除了登录页面的判断其他页面的登录状态。

​ 要实现这个功能，需要运用到express框架的中间件，注意：中间件是有顺序的，它是从上到下依次进行匹配的。所以这个代码要写到路由之前。

​ 修改app.js

//引入路由模块下面加上
//拦截请求，判断用户登录状态
app.use('/admin',(req, res, next)=>{
    //判断用户访问的是否是登录页面
    //判断用户的登录状态
    //如果用户是登录的，将请求放行
    //如果用户不是登录的，将请求重定向到登录页面
    if(req.url != '/login' && !req.session.username){
        res.redirect('/admin/login');
    }else{
        //用户是登录状态，将请求放行
        next();
    }
})

//引入加密模块
const bcrypt = require('bcrypt');
module.exports = async (req, res) => {
    //定义对象的验证规则
    const schema = {
        username: Joi.string().min(2).max(12).required().error(new Error('用户名不符合验证规则')),
        email: Joi.string().email().required().error(new Error('邮箱格式不符合要求')),
        password: Joi.string().regex(/^[a-zA-Z0-9]{3,30}$/).required().error(new Error('密码格式不符合要求')),
        role: Joi.string().valid('normal','admin').required().error(new Error('角色值非法')),
        state:Joi.number().valid(0,1).required().error(new Error('状态值非法'))
    };
    //异步实施验证
    try{
        await Joi.validate(req.body, schema);
    }catch(e){
        //验证没有通过
        // e.message;
        //重定向回用户添加页面
        //ES5语法
        //res.redirect('/admin/user-edit?message='+e.message);
        //ES6语法
        return res.redirect(`/admin/user-edit?message=${e.message}`);
    };
    //根据邮箱地址查询用户是否存在
    let user = await User.findOne({email: req.body.email});
    //如果用户已经存在，邮箱地址已经被别人占用
    if(user){
        //重定向回用户添加页面
        return res.redirect(`/admin/user-edit?message=邮箱地址已经被占用`);
    }
    //对密码进行加密处理
    //生成随机字符串
    const salt = await bcrypt.genSalt(10);
    //加密
    const password =await bcrypt.hash(req.body.password,salt);
    //替换密码
    req.body.password = password;
    //将用户信息添加到数据库中
    await User.create(req.body);
    //将页面重定向到用户列表页面
    res.redirect('/admin/user');
    // res.send(req.body);
}

3.数据分页功能

​ 当数据库中的数据非常多，数据需要分批次显示，这时就用到了数据分页功能。

要素：1.当前页，用户通过点击上一页或者下一页或者页码产生，客户端通过get参数方式传递到服务器端

2.总页数，根据总页数判断当前页是否为最后一页，根据判断结果做出相应操作

  • 总页数：Math.ceil(总数据条数/每页显示数据条数)

limit(2) //limit 限制查询数量，传入每页显示的数据数量

skip(1) //skip 跳过多少条数据，传入显示数据的开始位置

  • 数据开始查询位置 = （当前页-1）*每页显示的数据条数

修改userPage.js

//导入用户集合构造函数
const { User } = require('../../model/user');
module.exports = async (req, res)=>{
    //接受客户端传递过来的当前页参数
    let page = req.query.page||1;
    // res.send(page);
    // return;
    //每一页显示的数据条数
    let pagesize = 10;
    //查询用户数据的总数
    let count = await User.countDocuments({});
    // res.send('用户的总数是'+count);
    // return;
    //总页数,向上取整比如3.1，会显示4
    let total = Math.ceil(count/pagesize);
    // res.send('总页数是'+total);
    //页码对应的数据查询开始位置
    let start = (page-1)*pagesize;

    //将用户信息从数据库中查询出来
    let users = await User.find({}).limit(pagesize).skip(start);
    // res.send(users);
    //渲染用户列表模板
    res.render('admin/user',{
        users: users,
        page: page,
        total: total
    });
}
