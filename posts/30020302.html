项目-开发规范

编码规范

必须执行

【必须】eventBus用过之后要记得off

// good code
created() {
    this.$bus.on("refresh", () => {});
}

destroyed() {
    this.$bus.off("refresh");
}

【必须】对象的深拷贝，要用lodash提供的cloneDeep方法

// good code
const obj = this.$lodash.cloneDeep(value);

// bad code
Object.assign(value, {})

【必须】文件或文件夹必须采用 中划线命名规范

【必须】组件的 data 必须是一个函数。

// bad code
export default {
    data: {
        foo: 'bar'
    }
}

// good code
export default {
    data () {
        return {
            foo: 'bar'
        }
    }
}

【必须】Prop 定义应该尽量详细，至少需要指定其类型。

// bad code
props: ['status']

// good code
props: {
    status: String
}

// better code 
props: {
    status: {
    type: String,
    required: true,
    validator: function (value) {
      return [
        'syncing',
        'synced',
        'version-conflict',
        'error'
      ].indexOf(value) !== -1
    }
  }
}

【必须】在组件上总是必须用 key 配合 v-for，以便维护内部组件及其子树的状态。

// bad code
<ul>
    <li v-for="todo in todos">
    {{ todo.text }}
    </li>
</ul>

// good code
<ul>
    <li
        v-for="todo in todos"
        :key="todo.id">
        {{ todo.text }}
    </li>
</ul>

【必须】使用多个attributes的元素应该分为多行，每行使用一个attribute。

// bad code
<my-component foo="a" bar="c" baz="c" />

// good code
<my-component
    foo="a"
    bar="b"
    baz="c"
/>

【必须】组件模板应该仅仅包含简单的表达式，多个复杂的表达式应该使用计算属性或方法重构。

// bad code
{{
    fullName.split(' ').map(function (word){
        return word[0].toUpperCase() + word.slice(1)
    }).join(' ')
}}

// good code
{{ normallizedFullName }}

// In JavaScript
// The complex expression has been moved to a computed properties
computed: {
    normalizedFullName: function() {
        return this.split(' ').map(function (word) {
            return word[0].toUpperCase() + word.slice(1)
            }).join(' ')
    }
}

【必须】除直接使用v-bind绑定整个对象的场景外，一律使用@和：来绑定事件和数据

【必须】单文件组件中，总是按照template，script, style(scoped), style的顺序来排放，必须存在script标签

推荐

【推荐】文件或文件夹单词尽量不要超过三个

【推荐】除特殊含义的简写外，尽量使用英文全拼

【推荐】复杂的计算属性分割成多个计算属性

【推荐】在 v-if/v-if-else/v-else 中使用 key

【推荐】获取更新后的dom能用nextTick就使用nextTick

// good code
this.$nextTick(() => {
    this.progressWidth = this.$refs.progress.width * 100;
});

【推荐】使用forEach或者map来代替for循环，除非需要break或者continue

// good code
const arr = [1, 2, 3];
arr.map((item) => {
    console.log(item);
    return item;
});

【推荐】如果出现需要判断这个值（非数字）存不存在，不存在则赋其他的值的情况，推荐以下这种写法

// good code
const temp = obj && {};

避免

【避免】尽量避免出现 元素选择器 ，若必须出现，也不能同时影响超过3层html元素

禁止

【禁止】禁止隐性的父子组件通信，不能使用this.$parent或者this.$root

提交规范

Git仓库分支规范

1. 每个项目的仓库分支如下：

  • master，主分支，不允许提交，最新的代码活跃分支
  • dev，主分支，不允许提交，最新的开发活跃分支
  • qa，次要分支，不允许提交，测试环境分支（如果需要）
  • release-x.y 次要分支，不允许提交，小版本release的分支
  • bugfix-x.y.z，release修复分支
  • 个人分支dev-{commitAuthor}，自行创建
  • feature-{function}，通用的独立feature

2. 分支说明

  • master 主分支，用来保存最新的稳定版代码，生产环境使用此分支部署
  • dev 主开发分支，用于开发环境部署，不允许提交
  • qa 次要测试环境分支，用于测试环境部署，不允许提交
  • release-x.y 次要的版本分支，不允许提交，x为大版本号，y为小版本号。
  • bugfix-x.y.z release分支修复（几乎不会有历史版本的修复，除非大的版本变动），修复完成后将此分支合并入release-x.y
  • dev-{commitAuthor} 自行创建的开发分支，提交个人代码后穿件pr到dev分支
  • feature-{function} 通用的独立feature，可以单独开此分支
  • • feature-dev-{function} 开发的按模块划分的feature分支，可以单独开此分支

Git代码提交规范

1. 代码提交必须按照类型，不可混合在一起提交 (使用 npm run cm 提交)

  • feat 新功能–提交一个新功能
  • fix bug修复–修复一个bug
  • docs 文档–新增或者更新文档说明
  • style 格式–格式化代码等操作
  • refactor 重构–可能包含bug修复和新功能添加，但是要区分开，有可能是整个业务逻辑调整性的代码）
  • perf 性能–提升性能的代码修改（业务逻辑没有问题，提升新能的代码，和bug修复区分）
  • test 测试–添加或修改一个测试
  • chore 工具–开发工具变动，不会影响到src和test文件的变动，不会修改我们项目代码的变动（例如开发工具变动，）
  • build 构建–依赖变动
  • ci 自动化–脚本变更
  • revert 回滚–代码回退，请谨慎使用，并且只能操作自己分支

2. 代码提交具体分支规范

  • 开发人员必须提交到dev-{commitAuthor}分支
  • 开发人员每天必须至少从 dev 分支拉取一次代码 ， 推荐每天两次 (早会前拉群代码，夕会前提交代码)
  • 开发人员每天必须至少提交一次pr到所在模块的feature-dev分支，该pr必须从dev-{commitAuthor}分支创建，并自测完成后提交
  • 前后端组长，必须每天至少合并一次pr, 并进行相应的代码lint检查
  • 所有代码必须dev完全测试完成后才可以上qa分支，以测试人员标准为准 (至少需要保证高优先级的bug完全解决)，特殊情况除外
  • 所有代码在qa必须测试完成之后才可以创建release标签或分支，已测试人员标准为准(至少需要保证中优先级的bug完全解决)，特殊情况除外
  • 若生产环境存在紧急问题，可以直接从release分支fork一个hotfix-release-x.y-bugId 来紧急修复，并发布到预上线环境测试之后方可发布
  • release分支在预上线环境中做测试之后提交到master分支，可作为发布使用
  • 只有master分支能发布上线

组件库开发规范

必须执行

【必须】所有组件必须存在单元测试

【必须】所有公共组件中不能存在网络请求

【必须】所有组件必须存在完整的文档和属性要求，优先使用examples中的文档

【必须】所有组件样式均使用scss单独存放，统一编译和导出 (还未支持)

避免

【避免】所有组件尽量不使用vuex，如必须使用，需要写清楚明确需求

项目开发规范

必须

【必须】所有数据操作，全部使用this.$lodash 或者 Vue.lodash

【必须】所有页面全部放在 pages 中，可以使用路由参数，不可以被其他页面或其他组件引用

【必须】所有组件全部放在 components 中，不可以使用或改变路由参数，不可以被其他组件引用

【必须】所有vuex必须模块，若涉及多个模块交互的数据，可以选择专门为功能建一个模块

建议

【建议】页面登录，注册，首页等关键页面不建议使用懒加载路由

Icon引入规范

统一使用iconfont网站集中管理icon，直接使用svg图标

  1. 在X-undefined项目中将产品上传好的图标添加得帆云前端图标项目中

  1. 在得帆云前端图标项目中将图标的Font Class / Symbol 更改为在前端工程中使用的名字，方便其他人查看

  1. 在前端图标项目中下载使用，注意更改图标名字

原型

必须执行

【必须】确保一比一还原，严格按照原型开发

【必须】原型中的颜色都要在颜色变量文件中定义，不允许直接使用颜色

【必须】所有的删除操作，都需要进行二次确认，确认内容请主动与产品经理确认

【必须】涉及数据请求的功能，一定有loading和空页面，具体效果请主动与产品经理确认

【必须】所有的接口请求消息提示均有后端返回，前端只控制显示(全局统一处理)

【必须】所有的操作性icon，请添加pointer类（移入小手和颜色变化，已在全局定义）

【必须】对于不确定长度的数据，请主动与产品经理确认是否一行展示，若是请使用XEllipsis 组件
